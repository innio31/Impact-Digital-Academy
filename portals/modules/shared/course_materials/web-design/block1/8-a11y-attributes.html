1. Introduction to Web Forms

Web forms are the primary means of user interaction on the internet‚Äîthey enable users to log in, register, make purchases, submit feedback, and perform countless other tasks. However, poorly designed forms create barriers for many users, particularly those with disabilities or those using assistive technologies.

Why Form Attributes Matter

HTML form attributes do more than just add functionality‚Äîthey communicate essential information to:

  • Visual users: Through visible cues and instructions
  • Assistive technology users: Through semantic information that screen readers can interpret
  • Browser auto-fill features: Through proper naming and attributes
  • Validation systems: Through built-in browser validation

2. The Importance of Form Accessibility

The Scope of Web Accessibility

According to the World Health Organization, over 1 billion people worldwide live with some form of disability. Web accessibility ensures that these individuals can perceive, understand, navigate, and interact with the web.

Legal and Ethical Considerations

  • WCAG (Web Content Accessibility Guidelines): The international standard for web accessibility
  • Section 508: U.S. federal requirement for government websites
  • ADA (Americans with Disabilities Act): Has been interpreted to apply to websites
  • EAA (European Accessibility Act): EU requirements for accessibility

Benefits of Accessible Forms

  1. Broader audience reach
  2. Improved SEO (search engines favor accessible content)
  3. Better user experience for everyone
  4. Reduced legal risk
  5. Future-proofing (as accessibility standards evolve)

3. The <label> Element: The Foundation of Form Accessibility

The <label> element is the single most important element for form accessibility. It programmatically associates descriptive text with a form control.

Why Labels Are Critical

Screen readers announce the label when users focus on a form control. Without proper labeling, users might hear "Edit text" or "Checkbox" without any context about what information is expected.

Two Methods of Label Association

Method 1: Explicit Association (Recommended)

<!-- Using 'for' and 'id' attributes -->
<label for="username">Username:</label>
<input type="text" id="username" name="username">

<label for="terms">
  <input type="checkbox" id="terms" name="terms">
  I agree to the terms and conditions
</label>

How it works: The for attribute of the <label> must match the id attribute of the form control. This creates a programmatic link.

Method 2: Implicit Association

<!-- Wrapping the input inside the label -->
<label>
  Email address:
  <input type="email" name="email">
</label>

Note: While implicit association works, explicit association is generally preferred as it's more reliable across older browsers and assistive technologies.

Multiple Labels

<!-- AVOID: Multiple labels -->
<label for="age">Age:</label>
<label for="age">Your age in years:</label>
<input type="number" id="age" name="age">

<!-- BETTER: Combine the text -->
<label for="age">Age (in years):</label>
<input type="number" id="age" name="age">

Visually Hidden Labels

<label for="search" class="visually-hidden">Search the site</label>
<input type="search" id="search" name="search" placeholder="Search...">

.visually-hidden {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  border: 0;
}

Important: This is better than using display: none or visibility: hidden, which removes the label from the accessibility tree entirely.

Label Styling for Usability

label {
  display: block;
  margin-bottom: 0.5rem;
  font-weight: 500;
  color: #333;
  cursor: pointer; /* Shows it's clickable */
}

/* Style for checkboxes and radios - often inline */
label.checkbox-label,
label.radio-label {
  display: inline-block;
  margin-right: 1rem;
  font-weight: normal;
}

4. Essential Form Attributes

4.1 The placeholder Attribute

The placeholder attribute provides a brief hint to the user about what kind of information is expected in an input field.

Basic Syntax

<input type="text" placeholder="Enter your full name">
<input type="email" placeholder="youremail@example.com">
<textarea placeholder="Write your message here..."></textarea>

Important Accessibility Considerations

‚ö†Ô∏è CRITICAL: Placeholder is NOT a substitute for a label!

<!-- WRONG: No label, only placeholder -->
<input type="text" placeholder="Full Name">

<!-- RIGHT: Label + placeholder -->
<label for="fullname">Full Name:</label>
<input type="text" id="fullname" name="fullname" placeholder="e.g., John Smith">

Problems with Placeholders

  1. Disappearing text: Once users start typing, the hint disappears
  2. Low contrast: Often light gray, difficult to read
  3. Memory burden: Users must remember the hint after it disappears
  4. Screen reader limitations: Not all screen readers announce placeholders consistently
  5. Mobile issues: On mobile, placeholders may be truncated

Placeholder Best Practices

<!-- GOOD: Placeholder provides example, not the only instruction -->
<label for="phone">Phone number:</label>
<input type="tel" id="phone" name="phone" 
       placeholder="(555) 123-4567"
       pattern="[0-9]{3}-[0-9]{3}-[0-9]{4}"
       title="Format: 555-123-4567">

<!-- BETTER: Persistent helper text + placeholder -->
<label for="credit-card">Credit card number:</label>
<input type="text" id="credit-card" name="credit-card" 
       placeholder="1234 5678 9012 3456"
       maxlength="19">
<p class="helper-text">Enter the 16-digit number without spaces or dashes</p>

Placeholder Pseudo-class for Styling

/* Style placeholder text */
input::placeholder {
  color: #767676;
  opacity: 1; /* Override Firefox default */
  font-style: italic;
}

/* For better contrast, ensure at least 4.5:1 contrast ratio */
input::placeholder {
  color: #6c757d; /* Darker gray for accessibility */
}

/* Different styles for different states */
input:focus::placeholder {
  color: #adb5bd;
}

4.2 The required Attribute

The required attribute indicates that a form control must have a value before the form can be submitted.

Basic Syntax

<input type="text" id="username" name="username" required>
<select id="country" name="country" required>
  <option value="">Select your country</option>
  <option value="us">United States</option>
</select>
<textarea id="message" name="message" required></textarea>

Visual Indication of Required Fields

There are two common patterns:

Pattern 1: Asterisk (*) method

<label for="email">
  Email address:
  <span class="required-indicator" aria-hidden="true">*</span>
  <span class="visually-hidden">(required)</span>
</label>
<input type="email" id="email" name="email" required>

.required-indicator {
  color: #c00;
  font-size: 1.2em;
}

/* Or using pseudo-element */
label.required::after {
  content: " *";
  color: #c00;
}

Pattern 2: Explicit text method (more accessible)

<label for="email">
  Email address:
  <span class="required-badge">required</span>
</label>
<input type="email" id="email" name="email" required>

.required-badge {
  background: #f0f0f0;
  color: #333;
  font-size: 0.75em;
  padding: 0.2em 0.5em;
  border-radius: 3px;
  margin-left: 0.5em;
}

Browser Validation and the required Attribute

Modern browsers provide built-in validation for required fields:

<form>
  <label for="name">Name (required):</label>
  <input type="text" id="name" name="name" required>
  <button type="submit">Submit</button>
</form>

Browser behavior: The browser will prevent form submission and show an error message if a required field is empty.

Customizing Required Field Validation

<!-- Custom validation message -->
<input type="text" id="username" name="username" required
       oninvalid="this.setCustomValidity('Please enter your username')"
       oninput="this.setCustomValidity('')">

<!-- Better approach with JavaScript -->
<script>
document.getElementById('username').addEventListener('invalid', function(e) {
  e.target.setCustomValidity('Please enter your username');
});
document.getElementById('username').addEventListener('input', function(e) {
  e.target.setCustomValidity('');
});
</script>

Required Fields and Screen Readers

Screen readers announce that a field is required when the required attribute is present:

  • NVDA: "Username, edit, required"
  • JAWS: "Username, edit, required, invalid entry" (when empty)
  • VoiceOver: "Username, required, text field"

Grouping Required Fields with Fieldset

<fieldset>
  <legend>Contact Information (all fields required)</legend>
  
  <label for="name">Full name:</label>
  <input type="text" id="name" name="name" required>
  
  <label for="email">Email address:</label>
  <input type="email" id="email" name="email" required>
</fieldset>

4.3 The autofocus Attribute

The autofocus attribute specifies that an element should automatically receive focus when the page loads.

Basic Syntax

<input type="text" name="search" autofocus>
<textarea name="message" autofocus></textarea>
<button type="submit" autofocus>Submit</button>

‚ö†Ô∏è Important Accessibility Considerations

USE WITH CAUTION! The autofocus attribute can create significant accessibility problems:

  1. Screen reader users: Focus moves immediately, potentially skipping navigation and page introduction
  2. Keyboard users: Cannot use keyboard shortcuts to navigate away before focusing
  3. Motor impairment users: May accidentally trigger actions
  4. Cognitive disability users: May be disoriented by unexpected focus movement

When to Use autofocus

Appropriate uses:
- Search pages (like Google's homepage)
- Login forms on dedicated login pages
- Single-purpose pages (e.g., "contact us" page with one form)

Inappropriate uses:
- Multi-purpose pages with forms below the fold
- Pages with navigation users might need first
- Modal dialogs (focus should be managed within the dialog instead)

Better Alternatives to autofocus

Option 1: Focus management with JavaScript (more controlled)

// Focus only after user has seen the page for a moment
window.addEventListener('load', function() {
  setTimeout(function() {
    document.getElementById('search').focus();
  }, 500);
});

// Or focus when a specific section becomes visible
const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      document.getElementById('search').focus();
      observer.disconnect();
    }
  });
});

observer.observe(document.getElementById('search-section'));

Option 2: Provide a "skip to form" link

<a href="#main-form" class="skip-link">Skip to the contact form</a>

<!-- Later in the page -->
<form id="main-form">
  <!-- form fields -->
</form>

.skip-link {
  position: absolute;
  left: -9999px;
  top: 0;
  background: #333;
  color: white;
  padding: 1em;
}

.skip-link:focus {
  left: 0;
  z-index: 1000;
}

Multiple autofocus Elements

If multiple elements have the autofocus attribute, the first one in the DOM receives focus:

<!-- First name will receive focus, NOT email -->
<input type="text" name="firstname" autofocus>
<input type="email" name="email" autofocus>

4.4 The disabled Attribute

The disabled attribute makes a form control unusable and unclickable. Disabled controls cannot receive focus, and their values are not submitted with the form.

Basic Syntax

<input type="text" name="username" disabled>
<select name="country" disabled>
  <option value="us">United States</option>
</select>
<button type="submit" disabled>Submit</button>

Visual Appearance of Disabled Elements

Browsers typically apply default styling to disabled elements:

/* Default browser styling is often light gray */
input:disabled,
select:disabled,
textarea:disabled,
button:disabled {
  opacity: 0.5;
  cursor: not-allowed;
  background-color: #f5f5f5;
}

/* Custom disabled styling */
input:disabled {
  background-color: #e9ecef;
  color: #6c757d;
  border-color: #ced4da;
  pointer-events: none;
}

When to Use Disabled Fields

Appropriate uses:
- Fields that become available only after certain conditions are met
- Showing calculated fields that users shouldn't edit
- Temporarily unavailable options
- Confirmation pages where data is read-only

Inappropriate uses:
- Hiding fields from users (use hidden or display: none instead)
- Permanently unavailable options (don't show them at all)

Accessibility Concerns with Disabled Fields

‚ö†Ô∏è Important: Disabled fields are not focusable and are often ignored by screen readers. Users may never know they exist.

<!-- BETTER: Use readonly for information users should see but not edit -->
<label for="calculated-total">Total amount:</label>
<input type="text" id="calculated-total" value="$150.00" readonly>

<!-- For conditional fields, use ARIA to communicate state -->
<label for="shipping-address">Shipping address:</label>
<input type="text" id="shipping-address" 
       aria-describedby="address-note">
<p id="address-note" class="helper-text">
  (Complete billing information first to enable this field)
</p>

Dynamic Disabling with JavaScript

<label>
  <input type="checkbox" id="same-address"> Billing address same as shipping
</label>

<label for="billing-address">Billing address:</label>
<input type="text" id="billing-address">

<script>
document.getElementById('same-address').addEventListener('change', function(e) {
  const billingField = document.getElementById('billing-address');
  if (e.target.checked) {
    billingField.disabled = true;
    billingField.value = ''; // Clear value or copy from shipping
  } else {
    billingField.disabled = false;
  }
});
</script>

4.5 The readonly Attribute

The readonly attribute specifies that an input field is read-only, meaning the user cannot modify its value but can still focus on it and copy its content.

Basic Syntax

<input type="text" name="username" value="johndoe" readonly>
<textarea name="terms" readonly>Terms and conditions text...</textarea>

Visual Indication for Readonly Fields

input:read-only,
textarea:read-only {
  background-color: #f8f9fa;
  border-color: #dee2e6;
  cursor: default;
  font-style: italic;
}

/* Pseudo-class for styling readonly fields */
input:-moz-read-only { /* Firefox */
  background-color: #f8f9fa;
}

When to Use Readonly Fields

Common use cases:
- Displaying calculated values (e.g., shopping cart total)
- Showing user information in profile review pages
- Fields that are auto-populated (e.g., username after registration)
- Terms and conditions text areas
- System-generated codes or IDs

Real-World Example: Order Review

<div class="order-review">
  <h3>Review Your Order</h3>
  
  <fieldset>
    <legend>Shipping Information</legend>
    
    <label for="review-name">Name:</label>
    <input type="text" id="review-name" value="John Doe" readonly>
    
    <label for="review-address">Address:</label>
    <input type="text" id="review-address" value="123 Main St" readonly>
    
    <label for="review-city">City:</label>
    <input type="text" id="review-city" value="Anytown" readonly>
  </fieldset>
  
  <fieldset>
    <legend>Order Summary</legend>
    
    <label for="subtotal">Subtotal:</label>
    <input type="text" id="subtotal" value="$99.99" readonly>
    
    <label for="tax">Tax:</label>
    <input type="text" id="tax" value="$8.50" readonly>
    
    <label for="total">Total:</label>
    <input type="text" id="total" value="$108.49" readonly class="total-field">
  </fieldset>
  
  <button type="submit">Confirm Order</button>
  <button type="button" onclick="history.back()">Edit Order</button>
</div>

Readonly and Form Submission

Unlike disabled fields, readonly field values ARE submitted with the form:

<form action="/submit-order" method="post">
  <input type="hidden" name="product-id" value="12345">
  
  <label for="product-name">Product:</label>
  <input type="text" id="product-name" name="product-name" 
         value="Premium Widget" readonly>
  
  <label for="quantity">Quantity:</label>
  <input type="number" id="quantity" name="quantity" value="2" min="1">
  
  <!-- This readonly field's value WILL be submitted -->
  <!-- The hidden field also submits, but is invisible to users -->
</form>

Combining Readonly with JavaScript

<label for="coupon">Coupon code:</label>
<input type="text" id="coupon" name="coupon" 
       placeholder="Enter code" 
       onchange="applyCoupon(this)">

<label for="discount">Discount applied:</label>
<input type="text" id="discount" name="discount" readonly>

<script>
function applyCoupon(input) {
  const discountField = document.getElementById('discount');
  
  // Simulate coupon validation
  if (input.value.toUpperCase() === 'SAVE10') {
    discountField.value = '10% off';
  } else if (input.value.toUpperCase() === 'SAVE20') {
    discountField.value = '20% off';
  } else {
    discountField.value = 'No discount';
  }
}
</script>

5. Additional Accessibility Attributes

5.1 aria-required Attribute

For custom form controls or when you can't use the HTML5 required attribute:

<div class="custom-select" 
     role="listbox" 
     aria-required="true"
     aria-labelledby="country-label">
  <!-- custom select implementation -->
</div>

5.2 aria-describedby for Helper Text

Connect form controls with additional descriptive text:

<label for="password">Password:</label>
<input type="password" id="password" name="password" 
       aria-describedby="password-hint password-requirements">
<p id="password-hint">Choose a strong password</p>
<ul id="password-requirements" class="requirements">
  <li>At least 8 characters</li>
  <li>Include a number</li>
  <li>Include a special character</li>
</ul>

5.3 aria-invalid for Validation Errors

Indicate when a field has invalid input:

<label for="email">Email:</label>
<input type="email" id="email" name="email" 
       aria-invalid="true"
       aria-describedby="email-error">
<span id="email-error" class="error-message" role="alert">
  Please enter a valid email address
</span>

5.4 autocomplete Attribute

Help users fill forms faster and more accurately:

<form>
  <label for="full-name">Full name:</label>
  <input type="text" id="full-name" name="full-name" 
         autocomplete="name">
  
  <label for="email">Email:</label>
  <input type="email" id="email" name="email" 
         autocomplete="email">
  
  <label for="phone">Phone:</label>
  <input type="tel" id="phone" name="phone" 
         autocomplete="tel">
  
  <label for="address-line1">Street address:</label>
  <input type="text" id="address-line1" name="address-line1" 
         autocomplete="address-line1">
</form>

Common autocomplete values:
- name, given-name, family-name
- email
- tel, tel-country-code, tel-national
- address-line1, address-line2, address-level1 (state/province), postal-code
- cc-number, cc-exp, cc-csc (credit cards)
- username, current-password, new-password

5.5 inputmode Attribute

Specify the type of keyboard to show on mobile devices:

<!-- Numeric keyboard on mobile -->
<label for="zip">ZIP code:</label>
<input type="text" id="zip" name="zip" 
       inputmode="numeric" pattern="[0-9]*">

<!-- Email keyboard with @ and .com buttons -->
<label for="email">Email:</label>
<input type="email" id="email" name="email" 
       inputmode="email">

<!-- Numeric keyboard with only digits -->
<label for="phone">Phone:</label>
<input type="tel" id="phone" name="phone" 
       inputmode="tel">

// Real-time password validation
document.getElementById('reg-password').addEventListener('input', function(e) {

  if (/[a-zA-Z]/.test(password)) {
    letterReq.classList.add('valid');
  } else {
    letterReq.classList.remove('valid');
  }
});

<script>
function goToStep(stepNumber) {
  // Hide all steps
  document.querySelectorAll('.form-step').forEach(step => {
    step.hidden = true;
  });
  
  // Show selected step
  document.getElementById(`step-${stepNumber}`).hidden = false;
  
  // Update step indicator
  document.querySelectorAll('.step').forEach((step, index) => {
    if (index + 1 === stepNumber) {
      step.classList.add('active');
      step.setAttribute('aria-current', 'step');
    } else {
      step.classList.remove('active');
      step.removeAttribute('aria-current');
    }
  });
  
  // Set focus to the step heading for screen readers
  document.getElementById(`step-${stepNumber}-heading`).focus();
}
</script>

8. Testing Form Accessibility

Automated Testing Tools

  1. WAVE (Web Accessibility Evaluation Tool) - Browser extension that highlights accessibility issues, checks for missing labels, ARIA usage, and contrast
  2. axe DevTools - Comprehensive accessibility testing, integrates with browser developer tools
  3. Lighthouse - Built into Chrome DevTools, provides accessibility audits with recommendations

Manual Testing Checklist

Keyboard Testing

[ ] Can I navigate to all form controls using Tab key?
[ ] Is the focus order logical and follows visual layout?
[ ] Is there a visible focus indicator on all elements?
[ ] Can I select radio buttons and checkboxes with arrow keys?
[ ] Can I open select menus with Space/Enter keys?
[ ] Can I submit forms with Enter key?
[ ] Can I navigate out of form fields without getting stuck?

Screen Reader Testing

With NVDA (Windows) or VoiceOver (Mac):

[ ] Are all form controls properly labeled?
[ ] Does the screen reader announce required fields?
[ ] Are error messages announced when they appear?
[ ] Is the purpose of each field clear from the announcement?
[ ] Are fieldset/legend used correctly for groups?
[ ] Is ARIA-describedby content being read?

Color and Visual Testing

[ ] Is there sufficient color contrast (4.5:1 minimum)?
[ ] Is information conveyed by color also available in text?
[ ] Are error states visually distinct?
[ ] Are disabled fields still readable?
[ ] Is focus indicator clearly visible (not just color change)?

// Quick accessibility check in console
function checkFormAccessibility(formId) {
  const form = document.getElementById(formId);
  const inputs = form.querySelectorAll('input, select, textarea');
  
  inputs.forEach(input => {
    const id = input.id;
    const label = document.querySelector(`label[for="${id}"]`);
    
    if (!label && input.type !== 'hidden') {
      console.warn(`Input ${id || 'unnamed'} has no associated label`);
    }
    
    if (input.hasAttribute('required') && !input.getAttribute('aria-required')) {
      console.log(`Consider adding aria-required="true" to ${id || 'input'}`);
    }
  });
}

// Run the check
checkFormAccessibility('my-form');

9. Common Mistakes and How to Avoid Them

‚ñ∂ Click to see all 7 common mistakes (with wrong/right examples)

Mistake 1: Placeholder as Label

<!-- WRONG -->
<input type="text" placeholder="Full Name">

<!-- RIGHT -->
<label for="name">Full Name:</label>
<input type="text" id="name" placeholder="e.g., John Doe">

Mistake 2: Missing ID on Inputs

<!-- WRONG - label can't associate -->
<label for="email">Email:</label>
<input type="email" name="email">

<!-- RIGHT -->
<label for="email">Email:</label>
<input type="email" id="email" name="email">

Mistake 3: Improper Required Field Indication

<!-- WRONG - color alone -->
<label style="color: red;">Email:</label>

<!-- RIGHT - explicit indication -->
<label for="email">
  Email:
  <span class="required-badge">required</span>
</label>

Mistake 4: Forgetting to Group Radio Buttons

<!-- WRONG - separate fieldsets? no -->
<label>Male <input type="radio" name="gender" value="male"></label>
<label>Female <input type="radio" name="gender" value="female"></label>

<!-- RIGHT - grouped with fieldset -->
<fieldset>
  <legend>Gender:</legend>
  <label>Male <input type="radio" name="gender" value="male"></label>
  <label>Female <input type="radio" name="gender" value="female"></label>
</fieldset>

Mistake 5: Disabling Without Explanation

<!-- WRONG - no explanation -->
<input type="text" id="discount" disabled>

<!-- RIGHT - explanation provided -->
<label for="discount">Discount code:</label>
<input type="text" id="discount" readonly value="SAVE10">
<p class="helper-text">Discount applied automatically</p>

Mistake 6: Multiple Elements with Same ID

<!-- WRONG - duplicate IDs -->
<input type="text" id="address" name="shipping_address">
<input type="text" id="address" name="billing_address">

<!-- RIGHT - unique IDs -->
<input type="text" id="shipping-address" name="shipping_address">
<input type="text" id="billing-address" name="billing_address">

Mistake 7: Not Handling Focus Management

<!-- WRONG - focus lost on validation error -->
<form onsubmit="return validate()">
  <!-- form fields -->
</form>

<!-- RIGHT - focus moved to error -->
<script>
function validate() {
  if (!document.getElementById('email').value) {
    const errorDiv = document.getElementById('email-error');
    errorDiv.textContent = 'Email is required';
    errorDiv.setAttribute('role', 'alert');
    document.getElementById('email').focus();
    return false;
  }
  return true;
}
</script>

10. Best Practices Checklist

‚ñ∂ Click to expand full best practices checklist

For Every Form

Label Association

  • [ ] Every form control has an associated <label>
  • [ ] The for attribute matches the input's id
  • [ ] Labels are positioned near their controls (typically above or left)
  • [ ] Clicking on the label focuses the associated control

Required Fields

  • [ ] Use the required attribute for mandatory fields
  • [ ] Indicate required fields visually (asterisk or badge)
  • [ ] Include "(required)" in the label text for screen readers
  • [ ] Consider marking optional fields instead if most are required

Placeholder Usage

  • [ ] Placeholders provide examples, not labels
  • [ ] Sufficient color contrast for placeholder text
  • [ ] Helper text provided outside the input for important instructions
  • [ ] Never use placeholder alone without a label

Focus Management

  • [ ] Visible focus indicator on all focusable elements
  • [ ] Logical tab order following visual layout
  • [ ] autofocus used sparingly and appropriately
  • [ ] Focus moves to error messages when validation fails

Error Handling

  • [ ] Error messages clearly associated with fields (aria-describedby)
  • [ ] Error messages announced to screen readers (role="alert")
  • [ ] Fields with errors marked with aria-invalid="true"
  • [ ] Multiple errors listed clearly at top of form

Disabled and Readonly

  • [ ] disabled only used when fields shouldn't be submitted
  • [ ] readonly used for information users should see but not edit
  • [ ] Visual distinction between normal, disabled, and readonly states
  • [ ] Explanation provided for disabled fields when appropriate

Advanced Accessibility

ARIA Attributes When Needed

  • [ ] aria-required="true" for custom controls
  • [ ] aria-describedby connecting to helper text
  • [ ] aria-invalid for error states
  • [ ] aria-live regions for dynamic updates

Semantic HTML

  • [ ] <fieldset> and <legend> for related controls
  • [ ] <optgroup> for grouped select options
  • [ ] Proper heading structure within forms
  • [ ] Native HTML elements preferred over custom controls

Mobile Considerations

  • [ ] Appropriate inputmode for different input types
  • [ ] Touch targets at least 44x44px
  • [ ] No horizontal scrolling on mobile
  • [ ] Zoom enabled (no user-scalable=no)

Testing Checklist

  • [ ] Tested with keyboard only
  • [ ] Tested with screen reader (NVDA/VoiceOver)
  • [ ] Tested at 200% zoom
  • [ ] Tested with high contrast modes
  • [ ] Tested on multiple browsers and devices
  • [ ] Automated accessibility tools run (WAVE, axe)

Conclusion

Form accessibility is not just about compliance‚Äîit's about creating inclusive experiences that work for everyone. The attributes covered in this guide (placeholder, required, autofocus, disabled, readonly) and the proper use of the <label> element form the foundation of accessible form design.

Remember these key principles:

  1. Always use labels - They are non-negotiable for accessibility
  2. Provide context - Don't rely on placeholders alone
  3. Indicate requirements clearly - Both visually and programmatically
  4. Test with real users - Automated tools can't catch everything
  5. Think progressively - Start with a solid HTML foundation, then enhance

Deployment Workshop: Deploying a React Frontend to Vercel or Netlify and Connecting to a Live Backend Service

Workshop Overview

This hands-on workshop guides students through the complete process of deploying a React application to modern hosting platforms (Vercel or Netlify) and connecting it to a live backend service. By the end of this session, students will have a publicly accessible full-stack application and understand the essential concepts of modern web deployment.

Duration: 3 hours
Prerequisites: - A React application (can be the capstone project or a practice app)
- A backend API (either deployed separately or using a public API)
- Git and GitHub account
- Basic familiarity with the command line

Part 1: Preparing Your React Application for Deployment (45 minutes)

1.1 Understanding Production Builds

Before deployment, we must understand the difference between development and production modes :

Aspect Development Mode Production Mode
Purpose Debugging and development End-user experience
Code Size Larger, includes comments Minified and compressed
Error Messages Detailed debugging info User-friendly messages
Performance Slower, includes hot reload Optimized for speed

Key Concept: React applications run differently in development versus production. The development build includes debugging tools and comprehensive error messages that help during coding but would slow down the user experience in production .

1.2 Creating a Production Build

Hands-on Activity: Open your terminal in your React project directory and run:

# For Create React App projects
npm run build

# For Vite projects
npm run build

What happens during build:

  • JavaScript files are minified (unnecessary characters like whitespace and comments are removed)
  • Assets are hashed for caching (e.g., main.abc123.css instead of main.css)
  • A new folder called build (CRA) or dist (Vite) is created containing all optimized files

Instructor Note: Explain that this build folder contains everything needed to run the application—it's a self-contained static site that can be served by any web server.

1.3 Testing the Production Build Locally

Before deploying, test the production build to catch any issues:

# For Create React App
npx serve -s build

# For Vite
npx serve -s dist

Visit http://localhost:3000 (or the provided URL) to verify:

  • All pages load correctly
  • API calls work as expected
  • No console errors appear
  • Client-side routing functions properly

1.4 Environment Variables for Different Environments

One of the most critical steps is configuring environment variables :

Why environment variables?

  • Keep sensitive data (API keys, secrets) out of source code
  • Different values for development, staging, and production
  • Easy to update without code changes

In React applications, environment variables must be prefixed with REACT_APP_ (for CRA) or VITE_ (for Vite) to be accessible in the browser .

Example .env files:

# .env.development (local development)
REACT_APP_API_URL=http://localhost:5000/api

# .env.production (will be set in hosting platform)
REACT_APP_API_URL=https://my-backend-api.com/api

Access in code:

const API_URL = process.env.REACT_APP_API_URL || 'https://default-api.com';

1.5 Handling Client-Side Routing

Single-page applications (SPAs) using React Router need special handling :

The Problem: When a user visits https://yourapp.com/about directly, the server looks for an about folder/file, finds nothing, and returns a 404 error.

The Solution: Configure the hosting platform to serve index.html for all routes, letting React Router handle the routing client-side.

Part 2: Deploying to Vercel (45 minutes)

2.1 Introduction to Vercel

Vercel is a cloud platform optimized for frontend frameworks like React. It offers :

  • Zero-configuration deployments - automatically detects React projects
  • Global CDN - serves content from locations closest to users
  • Automatic HTTPS and SSL certificates
  • Preview deployments for every pull request
  • Serverless functions support (if needed)

2.2 Method 1: Deploy via Git Integration (Recommended)

This method connects your GitHub repository to Vercel for continuous deployment .

Step-by-Step Instructions:

  1. Push your code to GitHub (if not already done): 
    git init
git add .
git commit -m "Ready for deployment"
git remote add origin https://github.com/yourusername/your-repo.git
git push -u origin main
  2. Sign up/Login to Vercel at https://vercel.com
    - Click "Continue with GitHub" for easiest integration
  3. Import your repository :
    - Click "Add New Project" → "Import Git Repository"
    - Select your GitHub repository
    - Vercel will automatically detect your framework (Create React App, Vite, etc.)
  4. Configure build settings :
    - Framework Preset: Should auto-detect (Create React App, Vite, etc.)
    - Build Command: npm run build or yarn build (auto-filled)
    - Output Directory: build (for CRA) or dist (for Vite)
  5. Add Environment Variables :
    - Click "Environment Variables" section
    - Add each variable (e.g., REACT_APP_API_URL)
    - Choose environments (Production, Preview, Development)
  6. Click "Deploy" and wait for completion

Success! Your app is now live at https://your-project.vercel.app

2.3 Method 2: Deploy via Vercel CLI

For developers who prefer the command line :

# Install Vercel CLI globally
npm install -g vercel

# Navigate to your project
cd my-react-app

# Deploy (follow interactive prompts)
vercel

# For production deployment
vercel --prod

2.4 Configuring Environment Variables in Vercel

After deployment, you can manage environment variables in the dashboard :

  1. Go to your project dashboard
  2. Navigate to "Settings" → "Environment Variables"
  3. Add key-value pairs (e.g., REACT_APP_API_URL = https://api.example.com)
  4. Specify which environments they apply to (Production, Preview, Development)

2.5 Custom Domain Setup

To use your own domain :

  1. In project dashboard, go to "Domains"
  2. Enter your custom domain (e.g., myapp.com)
  3. Follow Vercel's instructions to update DNS records
  4. Wait for propagation (typically a few minutes to 24 hours)

2.6 Continuous Deployment with Git

Once connected to GitHub, every push to the main branch automatically triggers a production deployment . Each pull request generates a unique preview URL for testing .

Part 3: Deploying to Netlify (45 minutes)

3.1 Introduction to Netlify

Netlify is another popular platform for deploying static sites and JAMstack applications :

  • Git-based workflow with automatic deployments
  • Built-in CDN and HTTPS
  • Easy form handling and serverless functions
  • Branch deploys and deploy previews

3.2 Method 1: Deploy via Git Integration

Step-by-Step Instructions :

  1. Push your code to GitHub/GitLab/Bitbucket
  2. Sign up/Login to Netlify at https://netlify.com
  3. Import your repository:
    - Click "Add new site" → "Import an existing project"
    - Choose your Git provider and authorize access
    - Select your repository
  4. Configure build settings :
    - Build command: npm run build (auto-detected)
    - Publish directory: build (for CRA) or dist (for Vite)
  5. Add Environment Variables :
    - Click "Show advanced" before deploying, or
    - After deployment: Site settings → Build & deploy → Environment
    - Add variables like REACT_APP_API_URL
  6. Click "Deploy site"

3.3 Method 2: Deploy via Netlify CLI

# Install Netlify CLI
npm install -g netlify-cli

# Login to Netlify
netlify login

# Initialize and deploy
netlify init

# Or manual deploy
npm run build
netlify deploy --prod --dir=build

3.4 Handling Client-Side Routing in Netlify

Create a _redirects file in the public folder (CRA) or public directory (Vite) :

/*    /index.html    200

This tells Netlify to serve index.html for all routes, enabling React Router to work correctly .

Important: This file must be placed in the public folder so it gets copied to the build output.

3.5 Deploy Previews and Branch Deploys

Netlify automatically creates unique URLs for every branch and pull request :
- branch-name--site-name.netlify.app for branch deploys
- Preview URLs for pull requests to test changes before merging

Part 4: Connecting to a Live Backend Service (45 minutes)

4.1 Understanding the Full-Stack Architecture

A deployed full-stack application consists of :

User's Browser → Your React App (on Vercel/Netlify) → Backend API (on Render/Heroku/AWS)

The frontend and backend are separate applications communicating via HTTP requests.

4.2 Backend Deployment Options

Platform Best For Key Features
Render Express/Node.js APIs Free tier, auto SSL, GitHub integration
Railway Full-stack apps Simple pricing, database hosting
Heroku Traditional apps Mature platform, many add-ons
AWS Amplify AWS ecosystem Integrated with Lambda, Cognito

4.3 Setting Environment Variables for Backend URL

In your React code, you should have been using an environment variable for the API URL:

// src/utils/api.js
import axios from 'axios';

const API_URL = process.env.REACT_APP_API_URL || 'http://localhost:5000/api';

export const api = axios.create({
  baseURL: API_URL,
});

For deployment, set the production API URL:

On Vercel:
- Project Settings → Environment Variables → Add REACT_APP_API_URL with value https://your-backend.com/api

On Netlify:
- Site settings → Build & deploy → Environment → Add variable

4.4 Handling CORS (Cross-Origin Resource Sharing)

When your frontend and backend are on different domains, the backend must allow requests from your frontend domain .

On the backend (Express example):

const cors = require('cors');

// Allow specific origin
app.use(cors({
  origin: ['https://your-frontend.vercel.app', 'http://localhost:3000'],
  credentials: true // if using cookies
}));

// Or allow all (not recommended for production)
// app.use(cors());

Important: Never use origin: '*' with credentials: true. Always specify allowed origins in production.

4.5 Testing the Connected Application

After deploying both frontend and backend:

  1. Open browser console to check for errors
  2. Test all API-dependent features (login, data fetching, form submissions)
  3. Verify environment variables are correctly loaded: 
    // In browser console
console.log(process.env.REACT_APP_API_URL)
  4. Check Network tab to ensure API calls go to the correct URL

4.6 Complete Full-Stack Deployment Example

Scenario: A task manager app with React frontend and Express backend .

Backend deployed on Render:
- URL: https://task-manager-api.onrender.com
- Endpoints: /api/tasks (GET, POST), /api/tasks/:id (PATCH, DELETE)

Frontend deployed on Vercel with:
- Environment variable: REACT_APP_API_URL=https://task-manager-api.onrender.com/api
- Code uses: axios.get(`${process.env.REACT_APP_API_URL}/tasks`)

CORS configuration on backend:

app.use(cors({
  origin: ['https://task-manager.vercel.app', 'http://localhost:3000']
}));

Part 5: Best Practices and Troubleshooting (30 minutes)

5.1 Deployment Best Practices

Practice Why It Matters How to Implement
Always test production build locally Catches issues before deployment Use serve -s build locally
Use environment variables Keeps secrets secure, enables different configs Set in hosting dashboard, not in code
Enable HTTPS Security and SEO benefits Both Vercel and Netlify provide auto HTTPS
Optimize assets Faster loading, better user experience Use WebP images, lazy loading, code splitting
Monitor performance Track user experience Use Vercel Analytics or Netlify Analytics

5.2 Common Deployment Issues and Solutions

Issue 1: Blank page after deployment
- Check: Browser console for errors
- Solution: Often caused by incorrect public URL or routing configuration

Issue 2: API calls failing with 404
- Check: Network tab to see actual request URL
- Solution: Verify REACT_APP_API_URL is set correctly and doesn't have trailing slashes issues

Issue 3: "Invalid Host Header" error
- Check: Backend CORS configuration
- Solution: Add your frontend domain to CORS allowed origins

Issue 4: Routes don't work on refresh
- Check: 404 page appears
- Solution: Configure redirects/routing rules

Issue 5: Environment variables undefined
- Check: Variable prefix (REACT_APP_ or VITE_)
- Solution: Ensure correct prefix and that variables were added before build

5.3 Security Considerations

  • Never commit .env files with real secrets
  • Use environment variables in hosting platforms for all sensitive data
  • Implement rate limiting on backend to prevent abuse
  • Enable HTTPS (automatic on Vercel/Netlify)
  • Set secure CORS policies - don't allow all origins in production

5.4 Performance Optimization Tips

  • Enable gzip compression on your backend
  • Use CDN caching (automatic on both platforms)
  • Implement lazy loading for routes and heavy components
  • Optimize images before uploading
  • Monitor Core Web Vitals using platform analytics
